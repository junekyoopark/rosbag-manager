ROSBAG Manager
by JKP  ·  Unmanned Systems Research Group

Upload, convert, and visualise ROS bag files in the browser. Powered by Rerun for 3D playback.

How to use
1
Upload a bag

Click Upload in the top-right. Drag and drop a .bag, .mcap, .db3, or .zip file onto the drop zone (max {{ max_size_gb }} GB). The file uploads immediately and conversion starts automatically.

2
Wait for conversion

The progress bar tracks conversion steps: format detection → MCAP conversion (if needed) → RRD export → metadata extraction → thumbnail. ROS1 .bag files go through a ROS1 → MCAP → RRD pipeline. MCAP files convert directly to RRD.

3
Publish with metadata

After conversion a Publish form appears. Set the name, description, and tags, then click Publish. You can also leave it and edit later from the library.

4
View in the browser

Click any bag card in the library to open the Rerun web viewer in a modal. The Rerun viewer stays loaded as you browse and switch between bags — no reload lag. A progress bar shows the viewer initializing. Use the timeline at the bottom to scrub through time. Switch between the 3D, 2D, and Time series panels as needed.

5
Download and run locally

For large bags with lag in the browser, hover over a card and click ↓ RRD to download the recording. Then open it with the Rerun desktop app:

rerun downloaded.rrd

6
Open a local RRD without uploading

Click Local RRD in the navbar. Drop any .rrd file onto the page — it loads directly in the browser viewer with no server transfer.

7
Search and filter

Type in the search bar to filter by name, description, or tag in real time. Use the filter buttons below the navbar to show only Converting or Error bags. Click a tag chip on a card to filter by that tag. Select multiple tags and toggle Any / All to switch between OR and AND matching.

8
Toggle dark mode

Click the moon/sun icon in the navbar to switch between light and dark themes. The preference is saved to your browser and restored on each visit.

Supported formats

.mcap MCAP (ROS2 / standalone) MCAP → RRD (direct)
.db3 ROS2 SQLite bag SQLite → MCAP → RRD

3D view tips
No transform path / empty 3D view
The 3D view requires a TF frame tree. Click the 3D panel, then in the Blueprint panel on the right find Space origin and type your root frame (e.g. map, odom, or world). Entities attached to that frame or its children will appear.
Sensors missing in 3D even after setting origin
The bag is likely missing /tf_static, which carries fixed sensor-to-body transforms (camera extrinsics, lidar mount, etc.). Without it, Rerun cannot position those sensors in the world frame. Re-record the bag with /tf_static included, or use the 2D / time-series views instead.
Custom message types not shown
Rerun natively visualises standard sensor_msgs, geometry_msgs, and tf2_msgs types. Custom types (e.g. mavros_msgs, apriltag_msgs) are stored in the recording but not auto-visualised — they will appear as raw data in the streams panel.
Lag in browser viewer
Download the .rrd file and open it with the Rerun desktop app (rerun file.rrd) for full native performance. The desktop app handles large point clouds and long recordings much better than the browser.

Overview
Live View lets you stream live ROS topics from a robot directly in your browser via Lichtblick — an open-source Foxglove Studio fork. The server acts as a WebSocket relay, so you can monitor a robot even from a different network.
Robot (foxglove-bridge :8765) → Server nginx → Lichtblick in browser
Step 1 — Set up the robot

The robot needs foxglove-bridge running. It listens on port 8765 and does not need to know anything about this server. Install it once per robot.

ROS 2
# Install (once)
sudo apt install ros-${ROS_DISTRO}-foxglove-bridge

# Launch
ros2 run foxglove_bridge foxglove_bridge
ROS 1
# Install rosbridge (once)
sudo apt install ros-${ROS_DISTRO}-rosbridge-suite

# Launch  (default port: 9090 — use 8765 to match the default config)
roslaunch rosbridge_server rosbridge_websocket.launch port:=8765
Firewall — make sure port 8765 is open on the robot for incoming TCP connections from the server's IP. On Ubuntu: sudo ufw allow 8765/tcp
Auto-start on boot — add the launch command to a systemd service or ROS launch file so foxglove-bridge starts automatically when the robot powers on.
Custom port — if you use a different port, enter the full WebSocket URL when adding the robot: ws://192.168.1.50:9090
Step 2 — Add the robot in the server

Users with Robot Management permission (or admins) can add robots via the Robot Management page or directly from Live View.

1
Create a network

Go to Robot Management and add a network (e.g. Lab, Field Site A). Set the scan subnet to your LAN range (e.g. 192.168.1.0/24).

2
Find robots with LAN scan

Click Scan LAN next to the network. The server scans the subnet for hosts with port 8765 open (takes ~1 second). Click Use next to any found robot to pre-fill its WebSocket URL, then give it a name and click Add robot.

2b
Or add manually

If the robot is not on the same subnet as the server (e.g. it's on Tailscale), enter the URL directly: ws://100.x.x.x:8765 (Tailscale IP) or ws://robot-hostname:8765.

Step 3 — Manage Lichtblick layouts
1
Upload a layout (optional)

Click the + Upload layout button on the Live View page. Select a Lichtblick layout JSON file (exported from Lichtblick's Blueprint panel). Give it a name and upload. Layouts are shared across all users and robots.

2
Set a default layout per robot

If you have robot management permission, a layout dropdown appears on each robot's card. Select a layout to set it as that robot's default. When you open Lichtblick from that robot's button, the layout loads automatically. Leave it as "Default layout" to use the global selector.

3
Or select at open time

Before clicking a robot's Lichtblick button, use the Layout dropdown at the top of the Live View page to pick a layout for that session. If the robot has a default layout set, it takes priority over the global selector.

Step 4 — Connect from Lichtblick
1
Open Live View

Click Live View in the navbar. Registered robots appear grouped by network. A green dot means the robot is reachable from the server right now.

2
Click the right Lichtblick button

Each robot card has two buttons:

Lichtblick Direct — use when your browser is on the same LAN as the robot. Opens Lichtblick connected directly to ws://robot-ip:8765, bypassing the server entirely.

Lichtblick Proxy — use when accessing remotely (e.g. via Tailscale Funnel). The server relays the WebSocket connection to the robot over its local network.

Both buttons pre-load the selected layout automatically.

3
Copy URL manually (optional)

Use Copy direct or Copy proxy to copy just the WebSocket URL. Then in Lichtblick: Open connection → Foxglove WebSocket → paste → Open.

Troubleshooting
Robot shows offline (grey dot)
The server cannot reach port 8765 on the robot's IP. Check that foxglove-bridge is running, the IP is correct, and port 8765 is not blocked by a firewall. Try: nc -zv <robot-ip> 8765 from the server host.
LAN scan finds nothing
Verify the scan subnet matches the robot's actual LAN range. The server container scans via the host's network — if the subnet is wrong the scan hits the right IPs but nothing answers. Also confirm foxglove-bridge is running on the robot before scanning.
Lichtblick connects but shows no data
The connection is established but no topics are publishing. Check that your ROS nodes are running and publishing (ros2 topic list). Some topics only publish when sensors are active (e.g. cameras, lidar).
Proxy URL fails from remote
The proxy requires the server to reach the robot. If the server moved to a different network than the robot, the proxy can no longer relay — use Direct connection instead if your PC is on the same LAN as the robot.
wss:// vs ws://
When accessing this server over HTTPS (e.g. via Tailscale Funnel), the proxy URL uses wss:// (secure WebSocket). Browsers block mixed content, so you cannot use an insecure ws:// direct URL while on an HTTPS page — open Lichtblick in a separate tab (it's served at /lichtblick/) where the direct URL will work.

Prerequisites
Docker Engine 20.10+ and the Docker Compose v2 plugin (docker compose) must be installed on the host machine. Docker Desktop for Mac/Windows bundles both. On Linux: sudo apt install docker.io docker-compose-v2 or follow the official Docker Engine install guide.
Ports 80 (web UI) and 5555 (Flower task monitor) must be free on the host. If port 80 is in use, change the nginx port mapping in docker-compose.yml: e.g. "8080:80".
First-time setup
1
Clone the repository 
git clone <repo-url>
cd rosbag-manager
2
Create the environment file

Copy the example and edit the values:

cp .env.example .env
nano .env   # or any editor

Required values to set:

POSTGRES_PASSWORD — database password (any strong string)
SECRET_KEY — session signing key (generate with openssl rand -hex 32)
INITIAL_ADMIN_PASSWORD — password for the first admin account
DATA_DIR — host path where uploaded bags and RRDs are stored (default: ./data)
3
Build images and start all services 
docker compose up --build -d

On first run Docker pulls base images and builds the backend/worker/nginx images. This takes 2–5 minutes. The server is ready when docker compose ps shows all services as healthy.

4
Run database migrations 
docker compose exec backend alembic upgrade head

This creates all tables and applies any pending schema migrations. Run this once on first setup and again after each git pull.

5
Open the web UI

Navigate to http://<host-ip> in a browser. Log in with INITIAL_ADMIN_USERNAME (default: admin) and the password you set in .env.

Daily operation
▶
Start the server 
docker compose up -d
■
Stop the server

Stops all containers but keeps data volumes intact:

docker compose down
↺
Restart a single service 
docker compose restart backend
docker compose restart worker
≡
View logs 
docker compose logs -f             # all services
docker compose logs -f backend    # API only
docker compose logs -f worker     # conversion worker only
Updating the server
1
Pull latest code 
git pull
2
Rebuild changed images 
docker compose build backend worker
3
Restart with new images 
docker compose up -d

Compose replaces only the containers whose image changed.

4
Apply any new migrations 
docker compose exec backend alembic upgrade head
Templates, CSS, and JS are baked into the Docker image — changes require docker compose build backend && docker compose up -d backend, not just a browser refresh.
Data & backups
Uploaded bags, RRD files, thumbnails are stored on the host at the path set by DATA_DIR in .env (default ./data/). Back up this directory to preserve all recordings.
PostgreSQL metadata (bag names, tags, descriptions, user accounts) lives in the postgres_data Docker volume. Export with: 
docker compose exec postgres pg_dump -U rosbag rosbag_viewer > backup.sql
Removing all data — to wipe the database volume completely: 
docker compose down -v   # WARNING: deletes postgres_data volume
Bag files in DATA_DIR are on the host and are not affected by -v; delete them manually if needed.
